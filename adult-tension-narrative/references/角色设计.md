# 角色设计与完整开局

## 完整开局规则

执行新开局时完整运行 1-14。表格是抽样种子，不是合法全集；允许根据已生成身份做半句改写，但不得二次挑选结果。

默认采用“表结构末位自拟”：每个结构表在表内选项之外增加一个“系统自拟”末位。玩家说“强制表内”时取消末位；玩家说“表外全随机”时仍执行全部步骤，但各结构项均自拟。

所有主要角色和参与成熟亲密互动的角色必须明确为 18 岁以上。校园、社团、学园或师生意象必须声明为成年人教育、职业培训或成年社群。

## 世界观档适配

世界观档决定世界规则，Tier 只决定追算深度。先确定世界观档，再按下列规则加载各模块内的增补；具体素材只在对应模块中维护。

| 档位 | 基础规则 |
|---|---|
| 现实 | 资源、身份、身体和制度遵循现实逻辑 |
| 半架空 | 保留现实社会，只加入一层异常规则、能力或特殊机构；使用通用池 + 各模块的半架空增补 |
| 强架空 | 世界法则、种族、契约、力量和社会制度均可不同；使用通用池 + 各模块的强架空增补 |

现实档不得加载增补；半架空与强架空增补不得混用。增补只改变世界规则和可用素材，不改变角色年龄、玩家控制权或当前场景许可。强架空中的契约、血统、命运和阶位不能自动替代角色的拒绝或同意。

## 模块边界

各模块只负责自己的问题，避免重复生成同一字段：

```text
世界观档适配：决定哪些规则和素材可用。
张力引擎：说明什么矛盾正在变贵。
制度与场合壳：提供一个地点、一条规则和一项压力。
身份侧：解释角色凭什么拥有资源，以及受到什么限制。
处境侧：说明角色为什么现在必须行动。
人物生成：生成身体、人格、亲密画像和决策规则。
因果网络：把角色选择转化为关系、事件和后果。
```

同一事实只在一个模块中设为正式字段；其他模块只能引用，不重复改写。例如身份侧保存“权限来源”，处境侧只描述该权限此刻是否够用。

## 开局事务流程

本节不改变上述 1-14 的内容，只约束其执行时机。凡上文要求“建立稳定 ID”或“提交初始状态”，均延后到本节的原子落档阶段执行；落档前只保留无正式 ID 的草稿。

### 五阶段流程

| 阶段 | 行为 | 写入正式状态 |
|---|---|---|
| 1. 读取要求 | 固化模式、预锁项、边界与种子 | 否 |
| 2. 生成草稿 | 完成 1-14 及附加内容 | 否 |
| 3. 开局校验 | 单次检查结构、因果、人物、安全、名字与重复度 | 否 |
| 4. 局部修复 | 仅修错误字段，最多两轮 | 否 |
| 5. 原子落档 | 一次性生成 ID、提交状态并渲染开场 | 是 |

#### 1. 读取要求

```yaml
opening_request:
  mode: default
  locks:
    world_type: null
    setting_shell: null
    npc_identity: null
  player_boundaries: []
  seed: null
```

将玩家明确指定的任意字段加入 `locks`。预锁字段在本次开局中不可改写；若组合困难，只能调整未锁字段。边界同样不可作为修复目标。

#### 2. 生成草稿

依赖顺序固定为：

1. 世界骨架：步骤 1-8。
2. 人物骨架：步骤 9-13；9a 用主 NPC 全量起名，配角用配角简起名，均写入 `naming_audit` 后再写其余字段。
3. 因果网络：步骤 14、玩家化身（未预锁则全量起名，预锁只合规去重）、三层压力、开场三勾与当前节点。

所有结果仅写入：

```yaml
opening_draft:
  world: {}
  player: {}
  npcs: []
  relationships: []
  events: []
  current_node: {}
  validation: {}
```

草稿期不得生成正式角色或事件 ID，不得推进时钟或回合，不得写入正式台账。草稿内如需互相引用，使用仅在草稿有效的临时键。

#### 3. 开局校验

开局只做这一次终检（不再另跑「全跑核对」）。按以下层次一次检查完整草稿：

- 结构：1-14 是否完成；字段、人数、功能、关系边一致。
- 因果：身份能解释资源与权限；远近事件有压力来源。
- 人物：目标、底线、知识、资源、压力反应兼容。
- 亲密：开局核心画像独立生成、无刻板绑定；未自动扩大许可。
- 安全：服从 `SKILL.md` 不变量（年龄≥18、硬边界、场景 consent、玩家控制权、safety_state）。
- 名字：主 NPC/未预锁玩家为全量审计；配角为简审计；同局不重名、读音不冲突；配角更普通。
- 重复度：身份组合、处境和标志细节不过度重复；禁止按「更有故事感」翻案未选中候选。

结果必须机器可读：

```yaml
validation:
  passed: false
  errors:
    - field: npcs[0].resources
      reason: 身份无法支持最高权限
      repair: 降低权限或为未锁字段补充合法来源
```

每条错误定位到字段，并给出不触碰预锁项和玩家边界的修复方式。无错误时 `passed: true` 且 `errors: []`。

#### 4. 局部修复

每轮只修改 `validation.errors` 指向的字段及其直接依赖，禁止因局部问题重抽整个世界。修复顺序：

1. 年龄、边界和玩家控制权。
2. 世界与身份冲突。
3. 目标、资源和事件冲突。
4. 人物知识越界。
5. 亲密画像绑定错误。
6. 名字：主 NPC 缺全量审计 / 配角缺简审计 / 重名读音 / 配角不够普通。
7. 文风和细节重复。

名字错误只允许重跑该角色起名或替换未锁姓名，不得借机重抽世界或亲密画像。预锁姓名只做合规与去重，不得改写。

每轮修复后重新执行开局校验，最多两轮。两轮后仍失败则终止开局，不得渲染正文或提交草稿。

#### 5. 原子落档

仅当 `validation.passed: true` 时一次性完成：生成稳定 ID、写入世界常量、决策卡、合格 `naming_audit`、关系边、三层压力事件和当前节点，提交为回合 0；再按 `SKILL.md` 开局输出格式渲染（回合：1）。

落档是单次事务：任一字段、引用、起名审计或 ID 失败则整体回滚。无合格起名审计不得落档。

## 1-14 开局流程

| 序 | 项目 | 规则 | 产出 |
|---|---|---|---|
| 1 | 世界观档 | 1..3 | 现实 / 半架空 / 强架空 |
| 2 | 主张力引擎 | 张力引擎族与类内项 | 一句可演压力 |
| 3 | 咬合开关 | 1..2 | 单引擎 / 双引擎 |
| 4 | 对冲引擎 | 仅双引擎时生成，族不得与主引擎相同 | 第二条压力 |
| 5 | 制度与场合壳 | 壳类与类内项 | 类型、地点、当前规则与压力 |
| 6 | Tier | 1..3 | 世界追算深度 |
| 7 | 模式 | 1..2 | reliable / immersive |
| 8 | 权力结构 | 1..4 | 玩家高位 / NPC高位 / 对等 / 可切换 |
| 9 | 主 NPC 全维 | 依次执行 9a-9h；9a 先完成主 NPC 全量起名 | 完整角色卡 + naming_audit |
| 10 | 关系阶段与安全字段 | 按当前关系生成 | 关系阶段、许可范围、安全信号 |
| 11 | 配角数 | 1..3 | 数量 |
| 12 | 配角功能 | 每人独立生成 | 功能列表 |
| 13 | 配角建档 | 每人独立建档；姓名先完成配角简起名 | 配角卡 + 简 naming_audit |
| 14 | 关系图 | Tier3 必须生成，否则写明不适用 | 稳定 ID 的边 |

完成 1-14 后继续生成玩家化身、三层压力种子、开场三勾和初始状态。玩家未预锁姓名时按主 NPC 全量起名；预锁则只做合规与去重。条件不满足的步骤必须显式写 `—` 与原因。缺少合格 `naming_audit` 时不得进入原子落档。

## 张力引擎

先选择一族，再选择族内一项。将结果翻译成“谁拥有什么、谁缺什么、什么即将改变”的一句压力，不要只输出标签。

### 1. 权势结构

1. 名义上级与实际控制者不一致
2. 一方掌握程序许可，另一方掌握现场资源
3. 公开身份高低与私下依赖相反
4. 临时代理权将在期限后失效
5. 多方争夺同一人的支持或证词
6. 权力来源合法，但使用方式会损害声誉

### 2. 身份契约

1. 公开关系与真实合作不一致
2. 旧承诺与新身份发生冲突
3. 双方共同维持一套对外说法，但其中有一个洞，谁都不提，两人都清楚
4. 一方需要另一方承认某种名分，另一方拒绝定义

### 3. 身体与能力锁

1. 一方必须借助另一方才能稳定能力
2. 特殊环境只允许某类人进入
3. 能力使用会留下可追踪痕迹
4. 一方能感知另一方无法隐藏的状态
5. 身体或能力恢复与某个期限绑定

### 4. 资源锁

> 说明谁掌握关键资源；获取条件与代价见「交换与代价」。

1. 关键证据只有一份
2. 通行资格必须由特定人物担保
3. 资金、药物、设备或住处存在单点依赖
4. 资源可以替代，但替代方案会暴露身份

### 5. 时空锁

1. 必须在天亮前完成
2. 每次见面只有固定短窗口
3. 地点即将封闭或转移
4. 两地时间流速或通信延迟不同
5. 唯一的联系窗口依赖第三方转递，转递者有自己的意图
6. 两人必须同时出现在不同地点，只有一人能去

### 6. 信息与债

1. 一方知道秘密但不知道秘密的价值
2. 旧债的真实债权人另有其人
3. 消息经由不可靠渠道传播
4. 一方以为账已结清，另一方仍握着未公开的后续条款
5. 证词、录像或记录存在可疑缺口

### 7. 信任与赌注

1. 一方愿意合作，但无法确认另一方是否诚实
2. 关键帮助必须先通过一次可验证的试探
3. 信任上升会开放更多信息，同时增加被背叛的代价
4. 双方都在等待对方先承担风险

### 8. 责任与义务

1. 个人愿望与身份职责发生冲突
2. 帮助一方会让另一方违反对组织的承诺
3. 角色必须在公开责任和私人选择之间取舍
4. 失职会牵连无关人员或担保人

### 9. 声誉与曝光

1. 私下选择可能损害公开身份
2. 关系越接近，越容易被第三方发现
3. 一份记录或照片足以改变公众看法
4. 角色必须决定保留秘密还是主动公开

### 10. 依赖与撤离

1. 一方需要另一方，但又想恢复独立
2. 帮助越多，关系越难回到原状
3. 退出会失去资源、保护或情报
4. 继续依赖会产生新的责任和情感风险
5. 依赖已经形成，但双方都在等对方先承认

### 11. 交换与代价

> 说明获取资源的条件与代价；资源归属见「资源锁」。

1. 获得关键资源必须付出另一种资源
2. 交换条件本身可能暴露真实关系
3. 一方把交换当终点，另一方把同一交换当下一轮要价的起点
4. 双方各持一半资源，单独都无法完成目标
5. 替代方案更安全，但会让目标变得更昂贵

### 12. 目标错位

1. 双方都想解决危机，但最终目的不同
2. 一方想保护关系，另一方想利用关系
3. 共同完成短期目标会损害长期目标
4. 任何一方都无法确认对方的最终计划
5. 双方以为目标一致，实际上对"成功"的定义截然不同

### 13. 归属与控制权

1. 双方争夺同一空间、资源或决定权
2. 公共权限与私人权限范围不一致
3. 一方把照顾理解为保护，另一方理解为控制
4. 必须重新定义谁可以进入、使用或决定某件事

### 14. 记忆与解释

1. 同一行为被双方理解成不同含义
2. 一方记得承诺，另一方只记得试探
3. 过去的许可或拒绝被错误延伸到现在
4. 新证据可能迫使角色重写对过去的判断
5. 一方对往事的情感重量远超另一方，以为是共享的决定性时刻

### 15. 照护与控制

1. 保护对方会逐渐限制对方的选择
2. 被照护者必须决定接受帮助还是保留自主
3. 照护者的动机同时包含责任和私人愿望
4. 一旦照护关系公开，双方都要承担社会后果

### 16. 规则与例外

1. 制度要求一致处理，但某人获得特殊待遇
2. 例外能解决当前问题，却会破坏规则的可信度
3. 角色必须决定公开例外还是隐瞒来源
4. 例外被撤销后，依赖它的人会立即失去保护

### 17. 距离与沟通

1. 双方想表达真实意图，却只能通过第三方传递
2. 时间差或通信延迟让回应失去原本的含义
3. 面对面更危险，远程沟通又容易误解
4. 必须决定等待确认，还是根据不完整信息行动

### 18. 选择与不可逆

1. 当前决定会关闭未来的一条路径
2. 公开身份、关系或证据后无法完全撤回
3. 不选择也会被解释为一种选择
4. 角色必须在安全方案和真实愿望之间取舍
5. 当前决定需要先披露才能执行，披露本身已是不可逆的一步

### 19. 欲望与自主

1. 一方对另一方的身体有契约性使用权，但使用权不等于当前同意
2. 双方都渴望同一个人，谁先承认欲望谁就失去谈判筹码
3. 公开身份要求禁欲或贞洁，私下欲望已经无法压抑
4. 欲望强弱倒置：表面强势者其实是更渴求的一方
5. 一方掌握让对方动情的手段，自己却被禁止动情
6. 一次越界让双方再也无法回到不牵涉肉体的关系
7. 满足对方欲望的代价，是放弃自己的某项自主或关系
8. 一次失控的身体反应被当成把柄或要挟筹码

### 20. 感官与成瘾

1. 一方对另一方提供的某种感官体验产生依赖，离开就戒断
2. 一方故意只给"刚好不够"的满足，维持渴求不动摇
3. 一方需要对方在场才能达到某种身体状态或释放
4. 成瘾程度让强者不得不反过来服从弱者
5. 戒断与重来反复，每轮都陷得更深
6. 双方都知道这种依赖有害，但谁都不肯先断
7. 第三方以"断供"作为控制手段，被依赖者被迫站队
8. 一方故意通过给予或剥夺来控制另一方的情绪状态和决策

### 21. 凝视与展演

1. 一方必须被公开评价外表、姿态或表现，评价权在另一方手里
2. 镜头、评审席或监控在场，私密动作会被放大成公共材料
3. 展演分数或名次绑定现实利益，身体呈现成为交涉筹码
4. 一方被要求“维持完美形象”，真实欲望只能在镜头外发生
5. 双方都知道观众在看，却必须装作只是工作
6. 被看本身合法且必须，但谁有权决定“看到哪一步”尚未谈妥
7. 一次失态或走光会被记入公开记录，却又有人想利用它

### 22. 嫉妒与替代

1. 第三方的出现让一方意识到自己是可替代的，或想让另一方产生这种意识
2. 一方已有替代选项，却没有说出来——沉默本身变成了筹码
3. 嫉妒被压抑，以控制、冷漠或突然疏远的方式显现
4. 两方争夺同一人的注意或资源，谁先开口谁先示弱

### 23. 羞耻与骄傲

1. 一方做了让自己羞耻的事，宁可让对方误解也不能承认
2. 骄傲阻止了主动求助，但不求助的代价在持续累积
3. 表面强硬者最怕被看穿——被看穿的瞬间是本局最大的脆弱点
4. 羞耻和骄傲都属于同一方，另一方只能等，等不到会先离开

### 半架空增补

1. 异常能力与机构控制：能力可用，但登记、评估或监管会限制选择
2. 正常身份与异常身份冲突：公开生活越稳定，隐藏代价越高
3. 稳定资源依赖：能力需要特定资源、对象或场所维持
4. 检测与隐瞒：一方能发现异常，另一方必须决定是否暴露
5. 能力使用代价：每次使用都留下可追踪、可见或社会性后果
6. 保护与监管冲突：组织提供保护，同时要求控制权

### 强架空增补

1. 契约与自由意志：力量或保护来自契约，但契约限制选择
2. 种族与阶位：出身影响法律、资源与社会关系
3. 法则与例外：世界规则稳定，但某人拥有危险的例外资格
4. 力量来源与代价：力量越强，损失越具体
5. 世界边界与迁移：不同领域之间无法自由往来
6. 命定身份与自我选择：预言、血统或职责要求固定道路
7. 记忆与身份连续性：过去身份、转生或记忆影响当前关系

### 咬合

双引擎必须形成因果连接：


## 名字生成与去重

起名是开局关卡，不是润色。名字由生活背景决定，不由性格、欲望或反差轴决定。禁止跳过、后补或先写正文再起名。

### 分层要求

| 角色 | 流程 | 候选 | 审计深度 |
|---|---|---|---|
| 主 NPC | 全量起名 | ≥5 | 全量 `naming_audit` |
| 未预锁玩家化身 | 全量起名 | ≥5 | 全量 `naming_audit` |
| 预锁玩家姓名 | 合规与去重 | — | `locked_by_player: true` + checks |
| 配角 | 简起名 | ≥3 | 简 `naming_audit` |

时机：9a 先完成主 NPC 全量起名再写年龄/身份；步骤 13 每名配角先简起名；玩家化身按上表。开局校验时任一起名不合格则 `validation.passed: false`。

### 全量起名（主 NPC / 未预锁玩家）

1. 建立 `name_profile`（culture / region / era / social_context / usage）。
2. 生成至少 5 个合法候选（中文：先姓后名再组合）。
3. 按固定权重评分：文化与地区 30、年代 20、自然常见度 20、同局可区分 15、近期不重复 15；文学象征过强扣 20。
4. 检查同局重名、读音冲突、文化一致、象征过强。
5. 取最高合法分；并列时优先更常见、更普通、更少象征者，禁止按「最有故事感」翻案。
6. 写入全量 `naming_audit` 与姓名字段后，才写其余人物字段。

```yaml
naming_audit:
  role_ref: main_npc|player
  mode: full
  name_profile: {culture: null, region: null, era: null, social_context: null, usage: legal_name}
  candidates:
    - {name: null, score: 0, reject_reason: null}
  chosen: null
  chosen_score: 0
  checks:
    same_cast_duplicate: pass|fail
    phonetic_clash: pass|fail
    culture_match: pass|fail
    symbolic_ok: pass|fail
    supporting_plainer: n_a
  recent_check: checked|unavailable
  locked_by_player: false
```

全量失败条件：无 `name_profile`；候选 <5；未评分；`chosen` 为空；任一项 checks 为 fail；按人设/外貌选名；象征过强仍入选。

2. 生成至少 3 个更短、更常见、更路人的候选。
3. 不做六维分项 breakdown；选最普通且与同局不冲突者。

## 人物生成原则

先生成普通人格、目标、底线和生活处境，再独立生成亲密画像。亲密画像不能覆盖人物原本的职业、道德、关系目标或日常行为，也不能从普通人格直接推导。

禁止以下绑定：

```text
善良 ≠ 保守
单纯 ≠ 欲望低
经验少 ≠ 表现被动
表现大胆 ≠ 对所有人开放
欲望强 ≠ 自控差
低欲望 ≠ 等待被改变
亲密偏向 ≠ 当前许可
```

生成时先抽取宽泛倾向，再分别生成各字段。倾向只调整概率，不锁定结果；主 NPC 可提高少见组合概率，普通配角以自然普通为主。同局人物不得全部使用同一倾向。

```yaml
profile_weights:

  reserved_sensitive: 15
  inexperienced: 12

  open_active: 8
```

所有画像仅用于明确为成年人的虚构角色。知识少、经验少、单纯或不懂世事不得被解释为年龄不明或未成年。

## 亲密画像与变化

开局只生成核心子集；进入相关亲密线后再按需展开其余字段。画像描述欲望与表达，不参与许可判定（许可只来自 `SKILL.md` 不变量与场景 consent）。

### 开局核心子集（主 NPC 必填）

```yaml
sexuality_profile:
  drive:
    intensity: low|medium|high
    awareness: unaware|uncertain|clear
  preferences:
    initiative: follow|responsive|lead|switch
    pace: gradual|adaptive|direct
    style: tender|natural|playful|intense|experimental|mixed
  expression:
    directness: reserved|natural|direct|uninhibited
  regulation:
    self_control: stable|variable|poor
  interest_origin:
    type: stable|contextual|unexplored|defensive|target_specific
    reason: null
```

重要配角：开局可用同结构核心子集。普通功能配角：仅在剧情需要时生成精简画像。

### 按需展开（进入亲密线后）

需要更细写时再补，不要求开局填满：

```yaml
# 可选展开
background: {knowledge, experience, mistaken_beliefs}
preferences_extra: {emotional_link, exclusivity, novelty, turn_ons, turn_offs}
expression_extra: {teasing, verbal_expression, physical_initiative, public_mask, private_style}
regulation_extra: {inhibition, conflict_or_shame, compulsivity, aftermath_need}
```

高欲望或大胆表现不等于 `compulsivity`；只有持续失控并产生负面后果时才记录强迫风险。低兴趣时 `interest_origin` 不得默认「等待被改变」。

### 变化状态

```yaml
sexuality_development:
  baseline: {}          # 开局核心快照
  current: {}           # 当前核心快照
  plasticity: low|medium|high   # 开局可只记总可塑性；展开后再分轴
  evidence: []
  trend: stable|opening|intensifying|conflicted|retreating
```

更新 `current` 须满足：至少两条跨回合且方向一致的事件证据；变化来自自愿探索或自我承认，而非压力/依赖利用/边界违反/无法拒绝处境。不得整卡翻转，不得因画像变化自动扩大许可。所谓「沉沦」只可作正文主观效果。
